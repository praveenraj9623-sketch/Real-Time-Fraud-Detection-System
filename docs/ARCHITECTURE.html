Real-Time Fraud Detection System - Architecture

Portfolio architecture for a credit-card fraud monitor using feature engineering, XGBoost, Isolation Forest, threshold tuning, FastAPI scoring, Kafka/Spark streaming, MongoDB alert storage, and a Streamlit investigation dashboard.

Architecture Summary

The system has two operating paths. The full production-style path sends CSV transactions into Kafka, consumes them with Spark Structured Streaming, calls the FastAPI scoring service, and stores suspicious transactions in MongoDB. The Windows-friendly demo path uses the Streamlit sidebar to score real rows from data/raw/creditcard.csv and insert REVIEW, FLAGGED, and BLOCKED alerts directly into MongoDB.

Dataset: Kaggle creditcard.csv Features: Time, V1-V28, Amount, engineered features Champion: XGBoost in local fraud monitor Storage: MongoDB fraud_alerts Dashboard: Streamlit on port 8502

Core Components

Feature Engineering

src/processing/feature_engineering.py

Adds amount_log, hour_of_day, is_night_transaction, and amount_zscore. Saves processed parquet and feature statistics.

Batch Modeling

src/modeling/train_batch.py

Uses SMOTE for imbalance, trains XGBoost and Isolation Forest, logs metrics/artifacts with MLflow, saves model bundles and validation data.

Threshold Tuning

src/modeling/threshold_tuner.py

Evaluates precision, recall, F1, AUC, AUPRC, and business-cost tradeoffs across thresholds. Writes the active threshold artifacts.

FastAPI Scoring

src/api/main.py

Exposes /health, /score, and /batch-score. Applies inference-time features, returns probability, decision, risk score, SHAP drivers, and recommended action.

Streaming Simulation

src/streaming/

Publishes transactions to Kafka and consumes micro-batches through Spark. The consumer calls FastAPI and saves alerts to MongoDB.

Dashboard

app.py

Shows live alerts, analytics, model performance, and risk investigation. Uses MongoDB as the dashboard source of truth.

Processed data data/processed/fraud_features.parquet Engineered model-ready feature table. Can be regenerated from raw CSV.

Alert store MongoDB fraud_detection.fraud_alerts Stores scored suspicious transactions with decision, SHAP features, merchant, event time, and stored time. event_time_utc is used for analytics; stored_at_utc is insert time.

Important Design Choices

  • AUPRC matters more than accuracy because fraud is rare.
  • Threshold tuning is separated from model training so business policy can change without retraining.
  • Event time and stored time are separate. Event time powers demo analytics; stored time tracks database insert time.
  • Demo alerts still use real model probabilities. The sampler chooses rows for visual variety but does not fabricate fraud scores.
  • SHAP drivers are mathematical because V1-V28 are anonymized PCA features, not named business fields.
