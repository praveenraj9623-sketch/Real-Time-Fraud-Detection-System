Workflow Stages

1. Raw dataset setup

Put the Kaggle credit-card fraud file at data/raw/creditcard.csv. The expected columns are Time, V1 through V28, Amount, and Class.

2. Feature engineering

The project creates four extra fraud-focused features: amount_log, hour_of_day, is_night_transaction, and amount_zscore. The same logic is reused during API inference.

3. Batch model training

train_batch.py loads processed parquet, applies SMOTE to balance the minority class for XGBoost training, trains XGBoost and Isolation Forest, logs MLflow runs, saves metrics, plots, validation data, and model bundles.

4. Threshold tuning

threshold_tuner.py evaluates threshold candidates and stores active-threshold metrics. This is important because fraud systems usually choose thresholds by precision, recall, F1, or business cost rather than default 0.5.

5. Real-time scoring API

FastAPI receives one transaction or a batch of transactions, engineers the extra features, calls the model, applies the active threshold, and returns the final decision band.

6. Alert creation

Alerts are written to MongoDB when transactions are REVIEW, FLAGGED, or BLOCKED. Each alert contains the original transaction, score, risk level, decision, recommended action, merchant metadata, event time, stored time, and SHAP explanation.

7. Monitoring dashboard

Streamlit reads MongoDB and model artifacts to show live alerts, risk analytics, model performance, and investigation views.

The active block threshold is read from models/optimal_threshold.txt and training statistics. The dashboard model comparison now uses active-threshold metrics rather than stale training-threshold values.

FastAPI Workflow

EndpointInputOutputPurpose
GET /health None Model path, threshold, dataset/training statistics Checks whether the API can load artifacts and serve scoring requests.
POST /score One transaction with Time, V1-V28, and Amount Fraud probability, decision, risk score, SHAP features, recommended action Primary real-time scoring endpoint.
POST /batch-score List of transaction JSON objects Per-transaction scores and aggregate decision counts Useful for testing or batch scoring small lists.

Streaming Workflow

sequenceDiagram
  participant P as transaction_producer.py
  participant K as Kafka topic transactions
  participant S as Spark fraud_consumer.py
  participant A as FastAPI /score
  participant M as MongoDB fraud_alerts
  participant D as Streamlit dashboard

  P->>K: Publish transaction JSON with UUID, timestamp, merchant
  K->>S: Micro-batch read
  S->>A: Score each transaction
  A-->>S: Probability, decision, risk, SHAP
  S->>M: Insert REVIEW / FLAGGED / BLOCKED alerts
  D->>M: Read alerts and analytics

This path is the production-style architecture. For a simple Windows demo, the dashboard's Generate Demo Alerts button performs a shorter path: CSV rows -> model score -> MongoDB -> dashboard.

Known Limitations

  • The Kaggle dataset is anonymized, so V1-V28 cannot be explained as direct business features.
  • Small transaction amounts can be real fraud examples in this dataset, so the dashboard notes that small-dollar fraud rows are expected.
  • Kafka/Spark setup on Windows can be heavier than the local demo path.
  • The review zip excludes raw data and binary model files, so another LLM can review logic but cannot retrain without the dataset.
